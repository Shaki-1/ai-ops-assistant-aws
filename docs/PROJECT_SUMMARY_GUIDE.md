# Project Summary Guide

## What Was Built

AI Ops Assistant is a defensive operations training application that combines AI-assisted troubleshooting, server health visibility, incident simulation, role-based access, support tickets, timeline tracking, alerting, deployment automation, and recovery workflows.

It is designed to help junior administrators and DevOps learners practice incident response patterns without executing risky actions from the browser.

## Why It Was Built

The project demonstrates how AI can support operational troubleshooting while still requiring human review. It also exercises real deployment concerns:

- Authentication and role access
- Safe handling of private variables
- Nginx reverse proxying
- HTTPS issuance and fallback
- PM2 process management
- WebSocket live updates
- Terraform rebuilds
- GitHub Actions deploys
- File-backed data backup/restore

## Core Features

### Analyzer

The Log Analyzer accepts logs, command output, quick-check samples, and simulation scenario data. The backend classifies the input context so AI can distinguish real unknown command errors from diagnostic output generated by known tools.

### AI Troubleshooting

AI responses are structured into summary, severity, evidence, likely causes, recommended checks, safe commands, next action, and escalation criteria. The UI displays these in a readable result panel.

### Quick Checks

Quick diagnostic buttons provide realistic training samples for common operations topics such as Nginx, backend health, disk, memory, SSH, API latency, certificates, DNS, PM2, firewall posture, Node.js runtime, and recent service warnings.

### Resource Dashboard

The Resource Dashboard shows live metrics and runtime health for admins. Metrics update through WebSocket when possible and REST polling when WebSocket is unavailable.

### Alert Engine

The backend includes a lightweight in-memory alert engine for CPU, RAM, disk, latency, failed requests, and AI analysis failures. Admins can acknowledge alerts and ask AI to explain active alerts.

### Security Center

Security Center provides defensive checks for browser protocol, API reachability, metrics availability, auth/session state, host/domain, recommended ports, backend exposure, certificate/rate-limit reminders, and data handling. It avoids offensive scanning.

### Simulation Lab

Simulation Mode lets users practice incidents such as server down, high CPU, memory pressure, disk full, Nginx 502, backend down, SSH brute force, suspicious login activity, possible compromise, DNS failure, certificate failure, and slow API latency.

Simulation data is clearly labeled and does not change the real server.

### Tickets and Inbox

Users can submit tickets, feedback, and suggestions. Admins can view, reply, change status, mark as read, and delete tickets. Users can see their own submissions and admin replies. Unread badges use per-user read state.

### Incident Timeline

The timeline records sanitized activity such as login success, AI analysis, ticket actions, alert actions, simulation analysis, metrics AI analysis, and security review activity. It is file-backed JSON for now.

### AI Remediation Plans

After analysis, admins/users can request a remediation plan that includes immediate actions, verification commands, rollback plan, risk notes, and escalation criteria. Commands are guidance only and are not executed by the app.

### Governance & Data Use

The Governance page reminds users not to paste secrets, private keys, personal data, passwords, API keys, or confidential data. It supports local acknowledgement history and manual export.

## Architecture

The browser loads a static frontend through Nginx. REST API calls go to `/api/*` and WebSocket traffic goes to `/ws`. Nginx proxies both to the Node.js backend on localhost port `3000`. PM2 keeps the backend alive.

Terraform provisions EC2, auto-discovers Amazon Linux 2023 AMI/default networking, creates a security group, and runs user-data to install and configure the application. GitHub Actions can deploy updates over SSH.

## Deployment Pipeline

Deployment consists of:

1. Local private variables exported as `TF_VAR_*`
2. `scripts/create_tfvars_from_env.sh`
3. `terraform init`
4. `terraform validate`
5. `terraform plan`
6. `terraform apply`
7. EC2 user-data bootstrap
8. PM2 backend startup
9. Nginx HTTP baseline
10. DuckDNS update
11. Certbot HTTPS attempt when safe
12. Optional GitHub Actions deploy for code updates

## Monitoring and Alerting

Monitoring is intentionally lightweight:

- `/api/status` for basic health
- `/api/metrics` for protected operational metrics
- WebSocket metric broadcasts for admins
- Alert thresholds for common resource and API issues
- Dashboard panels for alerts and AI metrics insight

## AI Features

AI is used for:

- Log analysis
- Quick check diagnosis
- Metrics review
- Alert explanation
- Security posture explanation
- Remediation plan generation
- Incident report generation
- Safe command generation

AI output must be reviewed before operational use.

## Security and Governance

The project uses RBAC and protected endpoints to separate admin and limited user capabilities. It validates saved sessions with `/api/me` to avoid stale browser role state. WebSocket connections require a valid token and fall back to REST when unavailable.

No offensive security tooling is included. Security Center is defensive and informational.

## Backup and Restore

Tickets, timeline, and history are stored in JSON files. Scripts can export/import this data, but EC2-local backups do not survive `terraform destroy`. Archives must be copied off EC2 before destroy.

## Lessons Learned

- Bcrypt hashes should be generated on EC2 instead of passed through Terraform templates.
- Nginx variables must be escaped correctly in shell heredocs.
- HTTPS should be attempted only after DNS points to the current EC2 IP.
- HTTP baseline should remain working if Certbot fails.
- WebSocket should have REST fallback.
- Browser localStorage role values must be validated against backend truth.
- File-backed data needs an explicit export/import workflow before rebuilds.

## Current Limitations

- File-backed JSON is not a durable database.
- Terraform destroy removes EC2-local data.
- HTTPS depends on DNS propagation and Let's Encrypt rate limits.
- Quick Checks are training samples unless a real collector is added.
- JWT localStorage is acceptable for training but not ideal for production.
